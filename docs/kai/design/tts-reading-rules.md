## 0. 背景 — 症状と原因の対応

2026-07-05 第 1 回リハーサル（`docs/kai/stream-review/test-live.md`）で観測された症状と、
現行実装（旧 kai プロジェクト `/Volumes/ExSSD/apps/seiichi3141/kai` のルールベース変換
`packages/tts/src/converter.ts` の JS 移植）における原因:

| 症状                                             | 原因                                                                                                                                                                            |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 「kai」が「けーえーあい」                        | 読み辞書に `kai` が未登録。辞書にない英字列は一律 `<ALPHA VAL=>` タグ化 = 1 文字ずつスペルアウト読み                                                                            |
| 英単語全般がアルファベット読み                   | 同上（辞書 43 語のみ。旧プロジェクトの主経路だった LLM 変換が未移植）                                                                                                           |
| 「こんにちは」「では」の「は」が「わ」にならない | 助詞補正が kuromoji の**係助詞のみ**対象。感動詞「こんにちは」・単独接続詞「では」は素通り。※「今日は」「駅では」等の係助詞は現行実装で既に正しく「わ」になる（回帰テスト対象） |
| 一気読み（音節・区切りなし）                     | 単語境界 `/` の挿入処理が存在しない（kuromoji の読みをそのまま連結）                                                                                                            |
| 息継ぎがない                                     | 文中に読点ポーズを補う処理がない + speechd が文を隙間なく連続再生                                                                                                               |

旧プロジェクトには LLM 変換（主経路）・sanitize 後処理・静的バリデーション・
読み辞書の自動拡張・実 CLI 評価ハーネスの完成した資産があり、本書はそれらの移植を軸に、
不足分（助詞規則の拡張・息継ぎ）を新規設計として加える。
なお **`/` 区切りの挿入は旧プロジェクトでも LLM 生成のみが実績**で、ルールベースの
`/` 挿入は前例のない新規設計である（§5.3）。

## 1. 目的と責務

kai の発話テキスト（漢字かな交じりの日本語）を、聞き取りやすい AquesTalk10
音声記号列（koe）に変換する規則と実装を定める。「正しく読む」（固有名詞・英単語・
助詞）と「人らしく区切る」（単語境界・息継ぎ）の 2 軸で品質を定義する。

**やらないこと（非責務）:**

- 発話テキスト自体の内容整形（フルパスを言い換える等）— kai_narrator の責務（Issue #9）
- 無音期間の解消（ハートビート実況）— kai_narrator の責務（Issue #10）
- 字幕テキストの整形 — 字幕は生テキスト側を表示する現行仕様のまま
- 音声合成・再生そのもの — aquestalk_cli / speechd の既存責務

## 4. データモデル — koe 記法の正典ルール

AquesTalk10 が受け付ける音声記号列の制約（旧プロジェクト `docs/archive/aquestalk-spec.md`
= 公式仕様 v1.8 要約、および実機確認結果より）。**変換器の出力はこの全条件を満たすこと。**

### 4.1 許可される文字（allowlist — これが唯一の正典）

- 全角ひらがな `ぁ-ゖ`（例外: `づ` `ぢ` `ゔ` は禁止 → §4.2）
- 長音符 `ー`（直前が母音のときのみ。文頭・促音直後・句切記号直後は不可）
- 句切記号: `。`（文末・長ポーズ）`？`（疑問・音上昇）`、`（短ポーズ）`/`（区切り・ポーズなし）`;`（区切り・次を強調）
- アクセント記号 `'`（アクセント核。当面は使わない — §13）
- タグ: `<NUMK VAL=数値>`（桁読み）`<ALPHA VAL=英大文字>`（スペルアウト。**最終フォールバック専用**）
- 実機で合成可能な拡張音節: `てぃ でぃ とぅ どぅ ふぁ ふぃ ふぇ ふぉ`（置換不要 — 実機確認済み）

注: AquesTalk10 仕様上は `+`（複合語密着）も有効だが、本プロジェクトでは**使用しない**
（旧プロジェクトの sanitize は `;` `+` を除去しており allowlist と不整合だった。
本書は §4.1 を正典とし、Phase B の sanitize 移植時に **`;` は保持・`+` は除去**へ変更する）。

### 4.2 禁止文字と置換規則

| 入力                         | 置換                                                 | 根拠                                                                     |
| ---------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------ |
| `づ` / `ぢ`                  | `ず` / `じ`                                          | 音声記号列で未定義（実機確認済み。実装済み）                             |
| `ゔ`                         | ば行（`ゔぁ→ば` 等）                                 | 未対応。**現行 formatKoe は未処理で素通りする — PR-A1 で追加**           |
| `ぐぃ ぐぅ ぐぇ ぐぉ`        | `ぎ ぐ げ ご`                                        | 未対応。**同上 PR-A1 で追加**                                            |
| `！` `!`                     | `。`                                                 | 未対応。感嘆は長ポーズで代替（実装済み）                                 |
| ハイフン類 `-‐‑–—−`          | `、`                                                 | 素通しで合成失敗（実機確認済み。実装済み）                               |
| `・` `：` `:`                | `;`                                                  | 未対応（実装済み）                                                       |
| 括弧・鉤括弧                 | 内容ごと除去（`（）()`）/ 記号のみ除去（`「」『』`） | 未対応（実装済み）                                                       |
| `…` `...`                    | 除去                                                 | 未対応（実装済み）                                                       |
| 漢字・カタカナ・英数字の残渣 | 最終 allowlist で除去                                | 1 文字でも残るとその文全体が合成失敗し字幕のみ縮退になるため（実装済み） |

### 4.3 読みの規則

1. **固有名詞・技術用語は読み辞書が最優先**（§4.4）。辞書 → kuromoji → タグ化の順。
   **辞書置換の結果はアトミックな保護スパン**として扱い、後段（kuromoji 再解析・
   `/` 挿入）に内部を触らせない（§5.1 の中間表現。レビュー指摘 M1: ひらがな読みを
   素で kuromoji に戻すと「ぶろーどきゃすと→ぶろ/ー/どきゃ/す/と」のように再分割され
   読みが壊れることを実測済み）
2. **助詞の読み分け**（kuromoji=ipadic の品詞で確定的に判定できることを実測済み）:
   - 係助詞「は」→「わ」（実装済み。「今日は」「駅では」の は もこれで正しくなる）
   - 単独の接続詞「では」「それでは」→「でわ」「それでわ」（新規）
   - 感動詞「こんにちは」「こんばんは」→「こんにちわ」「こんばんわ」（新規・日本語読み辞書）
   - 格助詞「へ」→「え」（新規。旧プロジェクトにも無かった規則）
   - 「を」→「お」（新規。AquesTalk 公式推奨）
3. **英単語は「単語として」ひらがな読み**。辞書にある語は辞書の読み、
   ない語は `<ALPHA VAL=>` スペルアウトに落ちる（Phase A の限界。Phase B で解消）
4. **数値は `<NUMK VAL=>`**（桁読み）。小数・負数・`#42` は現行実装のとおり
5. **単語境界 `/` の挿入**（一気読み対策）: 主担当は **Phase B の LLM**（プロンプトで
   2〜5 モーラ目安の `/` 挿入を指示 — 旧プロジェクトで実績のある唯一の方式）。
   ルールベースでの `/` 挿入は保護スパン中間表現の導入が前提の新規設計であり、
   **Phase C として分離**する（§5.3）。Phase A では `/` を挿入しない
6. **息継ぎを補う**: 読点のない長文（目安 25 モーラ超）は、接続助詞
   （て・で・が・ので・から・けど）の直後に `、` を挿入する（トークン走査中に判定。
   保護スパン IR は不要）
7. **文末は必ず `。` または `？`**（現行どおり）

### 4.4 読み辞書（technical-terms.json の拡充）

- **最優先追加: `"kai": "かい"`**（キャラ名。現状最大の恥ずかしい読み間違い）
- 旧プロジェクト v4 プロンプトの頻出英単語を辞書化して移植:
  `resume→りじゅーむ、query→くえりー、mock→もっく、fork→ふぉーく、use→ゆーず、live→らいぶ、broadcast→ぶろーどきゃすと、update→あっぷでーと、session→せっしょん、callback→こーるばっく、handler→はんどらー、overlay→おーばーれい、store→すとあ` 等
- 配信頻出語を追加: `YouTube→ゆーちゅーぶ、Ubuntu→うぶんとぅ、Mac→まっく、Python→ぱいそん、bash→ばっしゅ、shell→しぇる、hermes→へるめす、Codex→こーでっくす、README→りーどみー、JSON→じぇいそん、YAML→やむる、HTTP→えいちてぃーてぃーぴー、SSH→えすえすえいち、VM→ぶいえむ` 等
- 日本語読み辞書（新設 `japanese-readings.json`）: kuromoji の reading が発話に不適な語。
  `こんにちは→こんにちわ、こんばんは→こんばんわ` から開始
- **大文字小文字の照合規則**（レビュー指摘 R5 反映）: 非区別照合は**英字 3 文字以上の
  エントリのみ**。2 文字以下の頭字語（`AI` `CI` `CD` `PR` 等）は完全一致のみ
  （一般語への誤マッチ防止）。`kai` / `Kai` / `KAI` は 3 文字なので非区別で「かい」

## 5. 処理フロー

### 5.1 Phase A（決定論・即効）

前提となる構造変更（レビュー指摘 M2）: 現行 `tokenizeTextSegment`
（converter.mjs:246-258）はトークンを即座に文字列連結して品詞情報を捨てている。
Phase A では辞書置換の出力を保護スパンとしてタグ同様に扱う**セグメント列の中間表現**
（`{type: "text"|"tag"|"protected", value, ...}` — 既存の tag 分割処理の拡張で足りる）
に改め、読点補完はトークン走査中に行う。

1. 読み辞書置換（技術用語 + 日本語読み。長い語優先、§4.4 の照合規則）。
   **置換結果は protected セグメントとして分離し、kuromoji に再解析させない**
2. 絵文字・制御文字・三点リーダー除去、数値・残存英字のタグ化（現行どおり）
3. text セグメントのみ kuromoji トークナイズ → 読み + 品詞を取得
4. 助詞読み分け（§4.3-2 の 5 規則）と読点補完（§4.3-6）をトークン走査中に適用
5. formatKoe: カナ→ひらがな、句切記号正規化、禁止文字置換（`ゔ`・`ぐ行` を追加 —
   §4.2）、最終 allowlist（現行どおり）

### 5.2 Phase B（LLM 主経路 + ルールベースフォールバック — 旧プロジェクト構成の移植）

1. sei-win LLM（`auxiliary` と同じ OpenAI 互換、`temperature: 0`、
   `enable_thinking: false` 必須）に旧 v3 系プロンプト（v4 の頻出エラー強化を統合、
   `{{TERMS_SECTION}}` にテキスト中の辞書語を動的注入）で koe を生成。
   **`/` 区切り（2〜5 モーラ目安）はこの経路が主担当**
2. `sanitizeLlmKoe` 移植: 正規表現後処理（**`/` は絶対に消さない**。§4.1 正典に合わせ
   `;` も保持に変更・`+` は除去。不完全タグ除去・促音/長音の後処理・禁止文字置換・
   Unicode 記号除去）
3. `applyParticleCorrection` 相当の安全網: LLM 出力に対しても係助詞「は」補正を適用
   （旧プロジェクトの二段構えに忠実）
4. `validateKoe` 移植: 静的バリデーション（タグ外の英数字・漢字・カタカナ・
   禁止記号 10 項目）。**違反が 1 つでもあれば Phase A 経路の出力を採用**
5. タイムアウト（目安 2.5s）・接続失敗も同様にフォールバック。発話は決して止めない
6. 未知語の自動辞書拡張（term-discovery 移植): 変換対象に辞書外英単語があれば
   fire-and-forget で LLM に読みを生成させ、ひらがな検証の上 JSON へ追記

### 5.3 Phase C（ルールベース `/` 挿入 — 保留・別設計）

Phase B の LLM 経路で `/` 品質が十分なら実装しない。フォールバック経路（LLM 不達時）の
一気読みが問題になる場合のみ着手する。その際は本書を改訂し、①保護スパン IR を最後まで
運ぶ設計 ②自立語境界の除外規則（サ変名詞+する・補助用言・複合名詞の連接はマージ）
③2〜5 モーラの束ねアルゴリズム、を具体化してから実装する（レビュー指摘 M1/M2/R1）。

### 5.4 speechd 側（息継ぎの仕上げ）

- 文単位再生の間に `sentence_gap_ms`（デフォルト 300ms）の無音を挟む

## 6. エラー処理・縮退

| 失敗モード                        | 検知              | 挙動                                                                        | 復旧                 |
| --------------------------------- | ----------------- | --------------------------------------------------------------------------- | -------------------- |
| kuromoji 初期化失敗               | Promise null      | 前処理のみで変換続行（現行どおり）                                          | プロセス再起動       |
| LLM 不達・タイムアウト（Phase B） | 2.5s タイムアウト | Phase A 経路へフォールバック                                                | 次の発話で自動再試行 |
| LLM 出力がバリデーション違反      | validateKoe       | Phase A 経路へフォールバック + 違反内容を warn ログ（評価データとして収集） | —                    |
| 合成失敗（CLI 非ゼロ）            | 既存              | 字幕のみ縮退（既存）+ 失敗テキストをログに記録し fixtures 化                | —                    |

**原則:** どの失敗でも発話パイプラインを止めない。品質は落としても声は出し続ける。

## 8. セキュリティ

- **秘匿情報:** 発話テキストは speechd 側で producer マスク済みの前提を維持。
  本変換器は新たな秘匿情報に触れない。LLM へ送るのは発話テキストのみ
- **入力の信頼境界:** 発話テキストは kai 自身の生成物（外部入力ではない）。
  LLM 出力は validateKoe の allowlist 検証を通してのみ採用（インジェクション面でも安全側）
- **ネットワーク:** LLM 呼び出しは Tailscale 内（sei-win）のみ。公開ポートなし

## 9. テスト・検証

- **ユニットテスト（node --test、aquestalk-server 内）:**
  - 症状の再現ケースを必須で含める: `kai→かい`（大文字小文字全変種）、
    `こんにちは→こんにちわ`、単独`では→でわ`、`へ→え`、`を→お`
  - 回帰ケース: `今日は→きょうわ`・`駅では→えきでわ`（係助詞 — 現行実装で既に正、
    壊さないことの確認）、`づ/ぢ` 置換、タグ内保護
  - 辞書保護スパン: `broadcast を update` → 「ぶろーどきゃすと」の内部に kuromoji
    再分割の痕跡（余計な `/` や読み崩れ）が無いこと
  - 照合規則: `AI` が `air` 等の一般語にマッチしないこと（R5 ガード）
  - 読点補完: 25 モーラ超・読点なしの文の接続助詞直後に `、` が入ること
  - `ゔ`・`ぐ行` の置換（R6）
  - validateKoe: 禁止 10 項目の検出と正常系の空配列（旧テスト移植）
  - sanitizeLlmKoe: 旧プロジェクトの約 60 ケースを移植（`/` 保持・タグ内保護を含む。
    `;` 保持への変更分はケースを書き換え、変更理由をテストコメントに残す）
  - LLM フォールバック: モックで不達/違反 → Phase A 出力になること
- **runtime acceptance（必須）:** Mac 実機で症状文を合成して耳で確認

```bash
# 症状の再現テキストでの実機確認（リハーサルで問題になった文型）
curl -s -X POST http://100.106.136.117:8890/synthesize \
  -d '{"text":"こんにちは、kai です。では PR #6 の broadcast.sh を update します"}'
# 期待: 「こんにちわ、かいです。でわ、ぴーあーる ろく の ぶろーどきゃすと…」相当の koe が
# NDJSON で返り、スペルアウト読み・「けーえーあい」が発生しないこと
```

- **評価ハーネス（Phase B）:** 旧 `koe-prompt-eval.ts` を移植。合成失敗・違反テキストを
  fixtures に蓄積し、プロンプト版ごとの成功率を実 CLI で定量比較してから切替

## 10. 実装手順（コーディングエージェント向け）
