# kai 基本設計 — 索引と書き方ガイド

このディレクトリは kai の基本設計書を置く。**読者はコーディングエージェント**（Claude Code / Codex / kai 自身）である。設計書は「エージェントが迷わず、確認質問なしで実装に着手できる」ことを合格基準とする。

## ドキュメント構成

```
docs/kai/
├── requirements.md          # 要件定義（What）。機能は F-xx で参照
├── mvp-plan.md              # MVP 計画（マイルストーン M0〜M5）
├── design/
│   ├── README.md            # 本ファイル（索引 + 書き方ガイド）
│   ├── 00-system.md         # システム基本設計（全体構成・共通規約。全コンポーネント設計の前提）
│   └── <component>.md       # コンポーネント基本設計（テンプレート準拠）
└── templates/
    └── component-design.md  # コンポーネント設計書テンプレート
```

- **要件（requirements.md）** = 何ができるべきか。アーキテクチャを拘束しない。
- **システム設計（00-system.md）** = 全体をどう組むか。コンポーネント分割・共通規約（イベントスキーマ、相関 ID、設定、秘匿情報、配置）はここが正典。
- **コンポーネント設計（<component>.md）** = 各部品をどう作るか。1 コンポーネント = 1 ファイル。

## 書き方の原則（コーディングエージェントが迷わないために）

1. **具体値で書く。** 「適切に」「柔軟に」「必要に応じて」を禁止。タイムアウトは秒数、ポートは番号、スキーマは JSON 例、パスはリポジトリ相対で書く。
2. **境界を明示する。**「やらないこと（非責務）」を必ず書く。エージェントのスコープクリープは境界の曖昧さから生まれる。
3. **変更ファイルを allowlist で示す。** 「変更してよいファイル」に列挙されていないファイルは触らない、と読める状態にする。コアファイル無改変の担保になる。
4. **検証をコピペ可能にする。** runtime acceptance の手順は実行可能なコマンドで書く。「動作確認する」ではなく `curl -X POST ... | jq .queued` のように。
5. **1 設計書 = 1 PR サイズの実装単位に分割する。** 収まらないなら「実装手順」で PR を区切る（目安: 変更 5 ファイル / 300 行）。
6. **該当なしも書く。** テンプレートのセクションは削除せず「なし」と明記する。「検討漏れ」と「該当なし」を区別できるようにする。
7. **要件とマイルストーンに紐付ける。** 満たす F-xx / M-x を冒頭に書く。トレーサビリティが振り返り（F-30）の材料になる。
8. **未決事項を隠さない。** 決まっていないことは「13. 未決事項」に書き、決め方（実装時検証 / オーナー判断）を添える。エージェントは未決事項を勝手に決めず、指示された方法で解消する。
9. **hermes の制約を毎回確認する。** プロンプトキャッシュ不可侵・コア無改変・Footprint Ladder。設計がこれらに触れる場合は該当セクションに明記する。
10. **正典の重複を作らない。** 共通規約（イベントエンベロープ、相関 ID、ポート表、秘匿マスク）は 00-system.md だけに書き、コンポーネント設計からは参照する。

## 設計 → 実装のワークフロー

1. オーナー / オーケストレーターがテンプレートからコンポーネント設計書を起こす（00-system.md の規約に準拠）
2. 設計レビュー（必要なら Opus / Codex の並行レビュー）→ ステータスを「レビュー済み」へ
3. コーディングエージェントに設計書のパスを渡して実装を依頼（Issue に設計書リンクを貼る）
4. 実装 PR には「11. 完了条件」のチェック結果と runtime acceptance の証跡を記載
5. 実装で設計から乖離した点は設計書に反映してからマージ（設計書が常に実態を表す）
