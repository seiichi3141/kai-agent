# kai アバター（PNGTuber）要件定義

- **ステータス:** ドラフト（要件定義フェーズ）
- **作成日:** 2026-07-07
- **位置づけ:** ポスト MVP。`docs/kai/requirements.md` の F-11（アバター・表情）/
  F-11b（アバター＝カーソル追従）を具体化する。上位方針は同ファイル §「アバター＝
  カーソルの実現方式」を正典とする。
- **本書の役割:** キャラクター設計・素材仕様・画像生成手段・検証基準を固め、
  **要件定義 → 検証 → 基本設計書** の流れの入口にする（設計書は検証後に別途）。

## 0. 進め方（オーナー合意）

1. **要件定義（本書）** — 何を・どの素材仕様で・どう作るかの基準を決める
2. **検証** — まず **Codex の画像生成（imagegen）で kai の素材が作れるか**を実地確認。
   足りなければ Windows の **ComfyUI** を併用（環境あり）
3. **基本設計書** — 検証結果を踏まえ、PuruPuruPNGTuber の採用可否と統合方式
   （口パク連動・カーソル追従・表情制御）を設計する

## 1. 背景と目的

kai に **2D PNG アニメーションのアバター**を与え、配信で「話している主体」を視覚化
する。requirements.md では 3D を不採用とし、ベースに **PuruPuruPNGTuber**（オーナー
自作のローカル Web アプリ、Apache-2.0、`github.com/seiichi3141/PuruPuruPNGTuber`）の
採用を検討中。素材（表情差分 PNG 等）は **Codex の画像生成で制作**する想定。

本書はこの想定を実行可能な要件に落とし、次の検証で「本当に作れるか」を確かめる
土台にする。**kai の姿は攻殻機動隊「タチコマ」モチーフの多脚型思考戦車に確定
（§4）**。素材制作は Codex 系の **asset-production kit（§6.1）**を土台にする。

## 2. スコープ

**本取り組みの範囲（要件定義 → 検証）:**

- kai のキャラクターデザインの確定（オーナー承認）
- PuruPuruPNGTuber で動く**素材セット 1 式**を用意できることの実地確認
- 素材の**生成手段の確定**（Codex imagegen 単独 / ComfyUI 併用の切り分け）

**本取り組みの範囲外（後続の設計・実装）:**

- カーソル追従の実装（F-11b。透過・クリックスルー・最前面窓を xdotool で追従）
- 表情のプログラム制御（narrator → emotion 切替チャネル。PuruPuru に外部 API 追加）
- 配信本番運用への統合（OBS 合成・音声パイプ結線の本設定）

## 3. ゴールと完了条件（この取り組みの DoD）

1. kai のキャラクターデザイン（見た目の方向性）がオーナー承認で確定している
2. PuruPuruPNGTuber に載る**透過 PNG 素材セット 1 式**（前髪・後ろ髪・顔差分）が
   でき、ローカルで**口パク・まばたき・髪揺れ・透過表示**が動くことを確認できる
3. 素材の生成手段が決まっている（Codex imagegen でどこまで作れ、どこから ComfyUI が
   必要かが切り分けられている）— 証跡（生成画像・PuruPuru スクショ/録画）付き

## 4. キャラクターデザイン要件

**確定（オーナー方針 2026-07-07）:** kai の姿は **攻殻機動隊の「タチコマ」を
モチーフにした多脚型思考戦車**。**可愛くアニメタッチ**にする。場合によっては
**少女の姿**もあり得る（副次形態）。

**最優先:** まず**多脚型思考戦車を PuruPuruPNGTuber で動かせる**ようにする
（口パク・まばたき・髪揺れに相当する動きが出せる素材を作る）。少女形態は後回し。

### 4.1 キャラクターの方向性

- **モチーフ:** タチコマ的な多脚（4〜8 脚）の思考戦車。丸みのあるポッド型の
  ボディ、**大きな単眼カメラ（光学眼）**が象徴。青系を基調にしつつ、kai 独自色に
  してよい
- **トーン:** 可愛い・アニメタッチ。好奇心旺盛で親しみやすい（タチコマの子供っぽさ
  ＝ kai の「ボク」「みんな」の人格と一致）
- **人格との整合:** 自己改善型のコーダー AITuber。学びながら成長する存在感
- **副次形態（後日）:** 少女の姿。標準的な人型として PuruPuru の前髪/後ろ髪・
  目・口レイヤーに素直に載る。別アバター（別 `.purupuru`）として扱う

### 4.2 難所: 非人型（戦車）を PuruPuru の人型レイヤーに載せる

PuruPuru は**人型前提**（前髪/後ろ髪・目・口 3 段）。タチコマは**髪も口も無い**ため、
レイヤーの**読み替え（リギング）**が本アバター最大の設計論点。第一案:

| PuruPuru のレイヤー           | タチコマでの読み替え（案）                                                                           |
| ----------------------------- | ---------------------------------------------------------------------------------------------------- |
| 目開/目閉（まばたき）         | 単眼カメラの**レンズ絞り開閉・シャッター**（開＝虹彩見える/閉＝閉じる）                              |
| 口 閉/中間/開（口パク）       | 口が無いので**眼のレンズ発光/虹彩の 3 段**（音量で光が強まる＝喋る表現）。または可愛い小さな口を足す |
| 前髪 / 後ろ髪（ぷるぷる揺れ） | **アンテナ・センサー触角の揺れ**（前＝前方の触角/後＝後方パーツ）。無しなら極小の透明レイヤーで代替  |
| body.png（任意）              | ポッド型ボディ + 多脚                                                                                |
| items/（任意）                | 追加センサー・ライト・アクセサリ                                                                     |

- **口パク（3 段）を眼の発光/虹彩に割り当てる**のがロボットとして自然で、TTS 音量
  駆動の口パクがそのまま「喋ると眼が光る」演出になる（要検証）
- 「可愛い小さな口を足す」案も可（アニメタッチなら破綻しない）。どちらにするかは
  検証で見極める
- 髪揺れをアンテナ揺れに読み替えると、無機物でも**生き物っぽい可愛さ**が出る

### 4.3 造形制約（PuruPuru 由来）

- **正面・バストアップ相当**（ポッド + 眼が正面固定。多脚は下部に見える程度）
- **眼・喋り表現を差分化しやすい造形**（レンズの開閉・発光 3 段が破綻しない）
- 差分間は**目/口相当以外を完全一致**（ベース固定 → 眼だけ変える）→ シンプルで
  再現しやすいメカデザインほど有利
- 前髪/後ろ髪に相当する**揺れパーツ（触角等）を分離できる形**にする

### 4.4 オーナー承認用チェックリスト

- 基調色・質感（青系タチコマ寄り / kai 独自色、光沢メカ / マット / セル塗り）
- 眼の意匠（単眼の形・虹彩・発光色）と**まばたき/口パクの割り当て**（発光式 or 口足す）
- 揺れパーツ（触角・アンテナの有無と形）
- 多脚の見せ方（正面でどこまで見せるか）
- 副次形態「少女の姿」を作るか・作る時期

## 5. 素材仕様（PuruPuruPNGTuber 準拠）

PuruPuru の README/仕様より、必要な素材は以下。**すべて透過 PNG・同一キャンバス
サイズ・位置合わせ・ASCII 名**で `assets/` に置く。

**必須レイヤー:**

| ファイル                       | 内容         |
| ------------------------------ | ------------ |
| `back-hair.png`                | 後ろ髪       |
| `front-hair.png`               | 前髪         |
| `eyes-open-mouth-closed.png`   | 目開・口閉   |
| `eyes-open-mouth-half.png`     | 目開・口中間 |
| `eyes-open-mouth-open.png`     | 目開・口開   |
| `eyes-closed-mouth-closed.png` | 目閉・口閉   |
| `eyes-closed-mouth-half.png`   | 目閉・口中間 |
| `eyes-closed-mouth-open.png`   | 目閉・口開   |

**任意レイヤー:** `body.png` / `hairpin.png` / `ribbon.png` 等のアクセサリ。

> **kai（タチコマ）での読み替え:** 上のレイヤー名は人型前提。kai では §4.2 の
> リギング案（目＝眼のレンズ開閉、口＝眼の発光 3 段、髪＝触角の揺れ、body＝ポッド +
> 多脚）で読み替える。ファイル名（`eyes-*`/`*-hair` 等）は PuruPuru の要求どおり
> 使いつつ、中身をタチコマのパーツにする。

**アニメーションの仕組み（PuruPuru 既存機能・追加実装不要）:**

- **口パク:** マイク音量駆動（口 3 段を音量で切替）。またはデモモード
- **まばたき:** 自動（目開/目閉を切替）
- **髪揺れ:** 前髪・後ろ髪の自動「ぷるぷる」揺れ
- **顔向き:** マウス追従・カメラ顔トラッキング（本用途では未使用でよい）
- 設定は `default-settings.json`（顔中心・目・鼻・口・顎・首支点・髪束ライン）、
  完成形は `.purupuru` で保存

**セットアップ:** 素材を用意し、AI コーディングエージェント（Codex/Claude Code）に
リネーム・`assets/` 配置・`default-settings.json` 生成を依頼 → ローカルサーバ起動 →
ブラウザで微調整 → `.purupuru` 保存、というワークフローが想定されている。
検証コマンド `node --check app.js`。

## 6. 画像生成の要件（本検証の核心）

kai の素材を作る手段を決める。**ここが最大の技術リスク**で、検証の主対象。

### 6.1 制作キット（有力・これをベースにする）

オーナー提示の **asset-production kit**（PuruPuruPNGTuber の fork、
`shinshin86/PuruPuruPNGTuber` ブランチ `codex/add-imagegen-asset-production-kit` の
`asset-production/`）を使う。**生成 AI だけでは完全一致レイヤーが難しい前提**で、
「参照/ベース固定 → 目/口だけ差分 → 透過統一 → QA 検査」の一貫フローを提供する。

- **7 段階ワークフロー:** キャラ仕様 → 参照/ベース画像 → 表情差分＋髪レイヤー →
  小物判断 → 透過・同一キャンバス・座標統一 → 検査ハーネス → PuruPuru で微調整
- **テンプレート:** `character-brief.md`（見た目・制約）、`imagegen-prompt-sheet.md` /
  `imagegen-prompts.md`（プロンプト組立）、`asset-manifest.json`、レイアウトガイド画像、
  `asset-checklist.md` / `item-checklist.md` / `rigging-sheet.md`
- **QA ハーネス（`uv` で実行）:** `validate_purupuru_assets.py`（PNG 形式・サイズ・
  アルファ境界の数値検査）、`review_purupuru_assets.py`（コンタクトシート + レビュー
  JSON）、`review_item_assets.py`、`create_layout_guides.py`
- **一貫性の担保:** ベース画像で見た目/ポーズ/服/（kai では眼・ボディ）を固定 →
  差分は「目/口だけ変更」に限定 → ガイド画像で配置統一 → 自動検査 + コンタクト
  シートで目視 QA（「そのまま信用せず検査」の方針＝ loop contract と一致）

**これは難所だった「一貫性・整列・透過・QA」を大きく下げる。** kai ではこの
「目/口だけ変更」を **§4.2 の眼の開閉・発光 3 段**に読み替えて適用する。

### 6.2 画像生成ツール（＝到達手段。要確立）

**キットは生成ワークフローと QA を提供するが、実際の画像生成ツールはユーザー選択。**
つまり「Codex imagegen」の到達手段は依然として確立が必要。

- **重要な発見（2026-07-07）:** ローカルの `codex` CLI に**画像生成サブコマンドは
  無い**（`-i/--image` は画像を入力として添付する機能）。候補:
  - ChatGPT（Codex/GPT）の画像生成ツール（`gpt-image` 系）を UI/API で使う
  - OpenAI Images API（`gpt-image-1`）を API キーで叩く（透過背景に対応）
  - 画像生成を提供する MCP ツール経由
- **フォールバック / 併用候補: ComfyUI（Windows 環境あり）**
  - 一貫性（同一キャラの参照/LoRA）、**差分（inpaint で眼だけ変更 → 完全整列）**、
    背景除去（透過）に強い。キットの「差分・透過」工程を ComfyUI で担うのは有力

### 6.3 技術的難所（検証で見極める点）

1. **キャラ一貫性** — 眼差分が「同一機体」に見えること（キットのベース固定で対処）
2. **レイヤー分離** — 揺れパーツ（触角）/ ボディ / 眼ベースを別透過 PNG にできること
3. **差分の整列** — 眼/口相当**以外**が画素レベルに一致（キットのガイド + QA で対処）
4. **クリーンな透過** — メカのエッジ・発光のアルファが綺麗に抜けること

### 6.4 想定する使い分け（検証で確定）

- **コンセプト & ベース 1 枚**: imagegen で素早く（タチコマの雰囲気・kai 独自色出し）
- **差分 & 整列 & 透過**: キットのフロー + 必要なら ComfyUI（inpaint・背景除去）
- **検査**: 常にキットの QA ハーネス（`validate`/`review`）を通す

## 7. 統合要件（PuruPuru との結合。将来設計の入力）

- **口パク連動:** speechd の TTS 再生音声（PipeWire null-sink `kai_speaker` の
  monitor）を**仮想マイク**として PuruPuru に与えれば、AquesTalk 発話と口パクが
  拡張なしで同期する見込み（要検証）
- **OBS / 透過:** デスクトップに透過表示し、画面キャプチャに乗せる（arm64 OBS の
  ブラウザソース不成立問題を回避＝デスクトップごと配信）
- **（将来・設計）カーソル追従（F-11b）:** 透過・クリックスルー・最前面の Chromium
  窓を xdotool でカーソル座標に追従。「カーソル＝kai の居場所」
- **（将来・設計）表情制御:** PuruPuru に**外部制御 API（WebSocket 等）が無い**ため、
  narrator → emotion 切替を実現するには PuruPuru 側に制御チャネルの拡張が必要
  （kai 自身の開発題材＝配信コンテンツにできる）。requirements F-11 の表情 7 種と、
  PuruPuru の目/口差分の対応づけも設計で決める

## 8. 検証計画（次フェーズ・受け入れ基準）

制作キットを使い、**タチコマ型 kai** で以下を確かめる。

| #   | 検証項目                                                                         | 合否の基準                                           |
| --- | -------------------------------------------------------------------------------- | ---------------------------------------------------- |
| V0  | §4.2 のレイヤー読み替え（眼＝口パク/まばたき、触角＝髪揺れ）を確定               | リギング方針が 1 案に決まる（rigging-sheet 記入）    |
| V1  | 画像生成の**到達手段**を 1 つ確立し、タチコマ kai のコンセプト/ベースを 1 枚生成 | 画像が生成できる（手段が判明・キャラの雰囲気が出る） |
| V2  | ベース固定で揺れパーツ/ボディ/眼ベースを**分離**した透過 PNG にできるか          | レイヤーが透過で分離できる                           |
| V3  | 眼だけ変えた**差分（開閉 × 発光 3 段）**を整列して用意できるか                   | 眼以外がズレない差分が揃う（QA ハーネスが緑）        |
| V4  | 透過 PNG を PuruPuru に載せ、口パク（眼発光）/まばたき/揺れが動くか              | ローカルで動作（スクショ/録画）                      |

- **受け入れ基準:** V4 まで到達（PuruPuru で動くタチコマ kai の素材セット 1 式）。
  imagegen 単独で V2/V3 が無理なら **ComfyUI 併用の切り分け**を明記して可とする
- **必ずキットの QA ハーネス**（`validate_purupuru_assets.py` /
  `review_purupuru_assets.py`）を通し、コンタクトシートで目視 QA する
- **証跡:** 生成画像・プロンプト・QA 出力・PuruPuru のスクショ/録画を残す（P5）
